## 1. 目的與範圍

### 1.1 目的
建立一套 Windows 桌面程式，使用 **Visual Studio .NET 2026 / C# WPF / .NET 10（x64）**，讀取**既有且已包含完整報表資料**的 **Access `.accdb`**，並以 **RDLC** 產生「**A4、一人一頁、三聯式繳費單**」報表，版面需依照 `3聯式繳費單樣版.pdf`。

### 1.2 範圍內
- 讀取既有 `.accdb` 的資料表/查詢（下稱「報表資料集」）。
- 依 `3聯式繳費單樣版.pdf` 排版，以 RDLC 產生繳費單頁面。
- 條碼以**條碼產生函式庫生成 Code39 條碼圖片**並嵌入（資料內容由 `.accdb` 提供，不在程式內重算）。
- 輸出：列印（印表機）與匯出檔案（建議 PDF；亦可視需求支援 EMF/圖片）。

### 1.3 範圍外（本版明確不做）
- 不從 CSV 匯入、不在程式內進行金額/條碼/檢查碼等演算法計算。
- 不產生/變更 `.accdb` 的商業欄位（僅讀取；必要時可寫入「列印紀錄」但不影響主資料）。
- 不依賴 Word/Excel。

## 2. 名詞與約定

- **三聯式繳費單**：同一張 A4 上分成三個區塊（第一聯/第二聯/第三聯），每位學生一頁。
- **三段式條碼**：依臺灣超商代收規範的三段條碼（程式僅負責「顯示/列印」，資料由 DB 提供）。
- **Code39**：條碼符號規格（symbology）。本系統以「條碼產生函式庫」輸出**條碼圖片**；是否需要在輸入內容前後加 `*` 由所選函式庫規格決定（若需要，則由程式在產生圖片時補上）。

## 4. 資料來源需求（Access `.accdb`）

### 4.1 資料來源型態
- 程式需支援指定：
  - `.accdb` 檔案路徑
  - 報表資料表或查詢名稱（Table/Query）
- 讀取方式：OLE DB（Microsoft.ACE.OLEDB.12.0）

### 4.2 報表替換欄位（由 `.accdb` 直接提供）
> 本專案 **不計算/不重算** 下列欄位；僅從既有 `.accdb` 讀取並填入 RDLC。  
> 若缺欄或型別不符，視為資料來源錯誤（阻擋輸出）。

**基本資訊**
1. 科班別（班級，string）
2. 繳款人（姓名，string）
3. 學號（string，固定 6 碼）
4. 座號（string 或 int）

**金額明細**
5. 學期午餐費用（int）
6. 新生訓練餐費（int）
7. 暑期輔導餐費（int）
8. 退前一學期餐費（int）
9. 超商代收費（int）
10. 應繳金額（int）
11. 金額（實繳金額，int）
12. 實繳金額中文（string）

**銷帳與條碼**
13. 繳費期限（string；建議以樣版需求呈現，例如「YYYY/MM/DD」或「MM/DD」）
14. 銷帳編號（string）
15. 第1段條碼（string）
16. 第2段條碼（string；依規格為完整內容，不允許補 0 或裁切）
17. 第3段條碼（string）

> 其他存在於 `.accdb` 的欄位（例如：建檔編號、檢查碼、各中間計算欄位）若未被 RDLC 版面使用，程式可忽略，但不得在程式中另行推導或覆寫上述 17 欄。


### 4.3 欄位名稱映射（Mapping）
本 SSD 以「**報表欄位名稱**」作為 RDLC 的**標準欄位名（Canonical Fields）**。  
若 `.accdb` 實際欄位名不同，程式必須提供映射（Mapping）以在載入後建立同名 `DataTable` 欄位供 RDLC 使用。

#### 4.3.1 標準欄位名（RDLC 使用）與建議對應
| RDLC 標準欄位名 | 建議對應的 `.accdb` 欄位名 | 備註 |
|---|---|---|
| 科班別 | 班級 |  |
| 繳款人 | 姓名 |  |
| 學號 | 學號 | 固定 6 碼（資料庫端已確保） |
| 座號 | 座號 |  |
| 學期午餐費用 | 學期午餐金額 |  |
| 新生訓練餐費 | 新生訓練午餐費 |  |
| 暑期輔導餐費 | 暑期輔導午餐費 |  |
| 退前一學期餐費 | 退前一學期午餐費 |  |
| 超商代收費 | 超商代收手續費 |  |
| 應繳金額 | 應繳金額 |  |
| 金額 | 實繳金額 | 報表顯示欄名為「金額」 |
| 實繳金額中文 | 實繳金額中文 |  |
| 繳費期限 | 繳費截止日 / 繳費期限 | 依你 `.accdb` 實際欄位為準 |
| 銷帳編號 | 原始銷帳編號 / 銷帳編號 | 依你 `.accdb` 實際欄位為準 |
| 第1段條碼 | 第1段條碼 | RDLC 顯示使用條碼圖片欄位（見 §5.3）；不在 RDLC 內以字型產生 |
| 第2段條碼 | 第2段條碼 | 同上 |
| 第3段條碼 | 第3段條碼 | 同上 |

#### 4.3.2 映射提供方式
- **選配** `fieldmap.json`：部署時允許以設定檔調整欄位名（避免重新編譯）。
- **或** 程式內常數映射：需可維護版本並清楚記錄變更。

> 不論使用哪種方式，程式不得對欄位內容做演算法重算；僅允許「欄位名對應」與「字串/數值格式化（顯示）」。

#### 4.3.3 動態 SQL 組建規則（確保 DataTable 欄位名一致）
程式需依據映射設定檔（或內建映射常數）**動態組建 SELECT 語法**，使用 `AS` 別名將 Access 實際欄位名轉換為 RDLC 標準欄位名。

**範例**：若 `fieldmap.json` 定義如下映射：
```json
{
  "科班別": "班級",
  "繳款人": "姓名",
  "學期午餐費用": "學期午餐金額",
  "金額": "實繳金額"
}
```

則程式應產生類似以下 SQL：
```sql
SELECT
  [班級] AS [科班別],
  [姓名] AS [繳款人],
  [學號],
  [座號],
  [學期午餐金額] AS [學期午餐費用],
  [實繳金額] AS [金額],
  ...
FROM [報表資料來源]
ORDER BY [班級], [座號], [學號]
```

**實作要點**：
1. 若 Access 欄位名與 RDLC 標準欄位名相同，則不需加 `AS`（如 `[學號]`）。
2. 所有欄位名需以方括號 `[]` 包裹，避免保留字或特殊字元問題。
3. 動態 SQL 組建後，讀入的 `DataTable.Columns` 名稱必須完全符合 RDLC DataSet 定義。
4. 若映射設定有誤（如 Access 無該欄位），於執行查詢時會拋出例外，程式需捕捉並於 Log 顯示明確錯誤訊息。

## 5. 報表輸出（RDLC）

### 5.1 頁面規格
- 紙張：A4（直向）
- 一人一頁：每筆資料強制換頁（Group Page Break）
- 三聯式版面：同頁三段區塊，版面參考 `3聯式繳費單樣版.pdf`

### 5.2 版面元素（必需）
- 第一聯：標題、學生資訊、金額明細、簽收/說明區（依樣版）
- 第二聯：標題、學生資訊、金額明細、收款/備註區（依樣版）
- 第三聯：包含三段條碼（垂直排列）及銷帳/金額資訊（依樣版）

> 版面位置、字級、框線、虛線分隔、欄位對齊，需以樣版為準；RDLC 版型檔（`.rdlc`）視為受控交付物。

### 5.3 條碼顯示規則（以圖片嵌入，不做計算）

#### 5.3.1 原則
- **不在 RDLC 內使用條碼字型**。
- 程式端讀取 `.accdb` 的條碼文字欄位（第1段/第2段/第3段），呼叫「條碼產生函式庫」生成 **Code39 條碼圖片（建議 PNG）**，再以 RDLC **Image** 元件呈現。
- 本版僅做「條碼文字 → 條碼圖片」的呈現轉換；**不重算** 條碼內容。

#### 5.3.2 RDLC DataSet 需增加的影像欄位（由程式組裝）
在保留既有 17 個替換欄位的前提下，程式需額外在 `DataTable` 加入下列三欄（僅供 RDLC 顯示）：

| RDLC 欄位名 | 型別 | 內容 | 備註 |
|---|---|---|---|
| 第1段條碼_Img | byte[] | Code39 條碼 PNG bytes | 由第1段條碼產生 |
| 第2段條碼_Img | byte[] | Code39 條碼 PNG bytes | 由第2段條碼產生 |
| 第3段條碼_Img | byte[] | Code39 條碼 PNG bytes | 由第3段條碼產生 |

RDLC Image 控制項設定：
- Image Source：Database
- MIME type：`image/png`
- Value：對應 `Fields!第X段條碼_Img.Value`

#### 5.3.3 Code39 起訖字元 `*` 的處理
- 若所選條碼函式庫要求輸入內容必須包含起訖 `*`，則由程式以 `*{value}*` 包裝後再生成圖片。
- 若函式庫自行處理起訖，則程式不得重複包裝（避免雙重 `*` 導致掃碼失敗）。
- 不論採哪種方式，**人類可讀碼**（若樣版需要顯示）一律以 `.accdb` 欄位原值輸出，不額外補 `*`（避免混淆）。

#### 5.3.4 條碼品質建議參數（為掃碼穩定性）
- 輸出解析度：以列印 300 DPI 等效為目標（由函式庫參數控制）。
- 條碼高度/寬度、Quiet Zone（左右留白）需符合樣版與超商掃碼需求；若樣版未明確，建議左右各至少 10 個 module 寬度的留白。
- 任何條碼圖片生成失敗 → 視為該筆資料不可輸出（阻擋整批或依 §8.3 的策略處理）。

## 6. UI 規格（WPF）


### 6.1 主要畫面元件
- **資料庫檔案**
  - 【選擇資料庫…】按鈕：開啟檔案選擇器（僅允許 `*.accdb`）
  - 顯示目前選取的 `.accdb` 路徑（可複製）
- **資料來源**
  - 【資料來源】下拉：列出可用 Table/Query（若無法列舉，提供文字輸入框作為替代）
- **匯出設定**
  - 【匯出資料夾…】按鈕：選擇匯出存檔路徑（Folder Picker）
  - 顯示目前匯出資料夾路徑（可複製）
- **動作按鈕**
- 【預覽】：產生暫存 PDF 並預覽（見 §6.5 預覽模式說明）
- 【匯出 PDF】：輸出 PDF 至指定匯出資料夾
- 【列印】：將報表送至印表機（依 RDLC/渲染器能力；若不支援可先輸出 PDF 後列印）
- 【結束程式】：安全釋放所有占用資源並關閉程式（見 §6.4）
- **處理記錄（Log）**
  - 多行唯讀文字區（可捲動），即時顯示處理過程
  - 每筆 log 建議格式：`yyyy-MM-dd HH:mm:ss.fff [LEVEL] message`

### 6.2 必需功能（Use Cases）
1. [UC-001] 使用者可自行選擇 `.accdb` 檔案作為資料來源。
2. [UC-002] 使用者可選擇資料來源（Table/Query）。
3. [UC-003] 使用者可設定匯出存檔路徑。
4. [UC-004] 系統可產生暫存 PDF 並預覽（依 §6.5 預覽模式）。
5. [UC-005] 系統可將報表匯出為 PDF 至使用者指定路徑。
6. [UC-006] 系統可列印報表（允許選擇印表機/份數等，或走 OS 預設列印流程）。
7. [UC-007] 系統必須顯示處理過程 log（至少包含：載入資料庫、讀取筆數、載入 RDLC、渲染/匯出、完成）。

### 6.3 例外處理 UI [REQ-UI-ERR]
- **未選擇資料庫**：禁用【預覽/匯出/列印】並提示「請先選擇 accdb」。
- **無法連線/缺少 ACE Provider**：顯示明確訊息（含位元需求 x64），並提供「如何安裝/修復」提示。
- **資料來源不存在或無權限**：顯示錯誤並寫入 log。
- **缺欄/型別不符**：顯示「缺少欄位清單」，並寫入 log（不得繼續輸出）。
- **條碼欄位為空或含不允許字元**：顯示「資料異常清單」，允許使用者取消輸出或略過異常筆（策略見 §8.3）。
- **未設定匯出資料夾或路徑不可寫入**：提示並禁用【匯出 PDF】。

### 6.4 【結束程式】資源釋放規範 [REQ-UI-EXIT]
按下【結束程式】時，系統必須依序：
1. 停止任何進行中的匯出/列印工作（若可取消）。
2. 釋放 RDLC/渲染相關物件（Report / DataSource / Stream）。
3. 關閉並釋放所有資料庫連線（`OleDbConnection.Dispose()`）、資料讀取器與檔案串流。
4. 釋放任何暫存檔案鎖定（確保匯出資料夾可被刪除/移動）。
5. 寫入最後一筆 log：`[INFO] Application shutdown.`
6. 呼叫 `Application.Current.Shutdown()` 結束程式。

### 6.5 預覽模式規範 [REQ-UI-PREVIEW]
【預覽】功能定義為：**產生暫存 PDF 並開啟供使用者檢視**。系統可採用下列任一實作方式：

#### 6.5.1 方式一：系統預設 PDF 閱讀器（建議）
1. 將 RDLC 渲染為 PDF，存放於系統暫存資料夾（`Path.GetTempPath()`）。
2. 檔名建議：`LunchBill_Preview_{yyyyMMdd_HHmmss}.pdf`
3. 使用 `Process.Start()` 以系統預設 PDF 閱讀器開啟暫存檔案。
4. 暫存檔案於程式結束時清理（見 §6.4）。

**優點**：實作簡單、無額外相依性、使用者可使用熟悉的 PDF 閱讀器。  
**缺點**：開啟外部程式，非內嵌預覽體驗。

#### 6.5.2 方式二：WebView2 內嵌預覽
1. 將 RDLC 渲染為 PDF，存放於系統暫存資料夾。
2. 使用 WPF 的 **WebView2** 控制項載入暫存 PDF 檔案。
3. WebView2 需安裝 **Microsoft.Web.WebView2** NuGet 套件。
4. 需確保目標環境已安裝 **WebView2 Runtime**（Windows 11 內建；Windows 10 可能需額外安裝）。
5. 暫存檔案於程式結束時或切換資料來源時清理。

**優點**：內嵌於主視窗，使用者體驗一致。  
**缺點**：需額外相依性（WebView2 NuGet + Runtime）、增加部署複雜度。

#### 6.5.3 預覽模式選擇建議
- **預設採用方式一**（系統預設 PDF 閱讀器），以降低部署複雜度。
- 若需求明確要求內嵌預覽，則採用方式二（WebView2）。
- 無論採用哪種方式，預覽功能必須：
  - 在 Log 記錄預覽開始與完成（含暫存檔路徑）。
  - 若 PDF 產生失敗，顯示錯誤訊息並寫入 Log。
  - 若外部開啟失敗（方式一）或 WebView2 載入失敗（方式二），顯示錯誤訊息。

## 8. 資料驗證規則（僅驗證，不計算）

### 8.1 必填檢核 [REQ-VAL-001]
- 班級、建檔編號、學號、姓名不得為空
- 三段條碼、檢查碼不得為空

### 8.2 格式檢核 [REQ-VAL-002]
- 建檔編號：必須 5 碼（建議全數字；若 DB 已保證可降為長度檢查）
- 學號：必須 6 碼
- 繳費截止日：必須 4 碼（MMDD）
- 檢查碼：必須 2 碼
- 第1段/第2段/第3段條碼：不得為空白；內容是否允許包含 `*` 依條碼函式庫需求（預設不含，若需由程式補上）

> 條碼內容允許字元是否僅限英數，依你 DB 的既有資料與超商規範決定；本版預設「英數可印」。

### 8.3 異常資料處理策略 [REQ-VAL-003]
- **預設**：任何一筆資料檢核失敗 → 整批停止輸出，並列出錯誤明細（建議做法，避免產出不可繳費的單據）
- **選配**：提供「略過錯誤筆」模式（輸出其餘正常筆），但必須列出略過清單並要求使用者確認

## 9. 條碼（Code39）與掃描品質規範

> 本系統**不負責重新計算**條碼內容（第1/2/3段條碼），僅依既有 Access 資料欄位輸出。  
> 但為確保臺灣超商代收設備可順利讀取，本章規範 **Code39 條碼影像產生與列印品質**。

### 9.1 Code39 編碼規範（影像產生必須符合）[REQ-BC-001]
1. **條碼制式**：Code 39（標準 Code39；除非超商端明確允許，否則不使用 Extended Code39）。
2. **起始/結束字元**：輸出影像時，資料字串必須加上起訖 `*`：  
   - `*{第1段條碼}*`、`*{第2段條碼}*`、`*{第3段條碼}*`
3. **允許字元（標準 Code39）**：`0-9`, `A-Z`, `-`, `.`, 空白, `$`, `/`, `+`, `%`。  
   - 若 Access 欄位含不在集合內的字元，視為**資料異常**（不得自行替換），必須在 Log 顯示錯誤並依規格中止輸出（可設計為：略過該筆/中止整批）。
4. **資料長度**：單段條碼（不含 `*`）不得超過 20 字元；本案三段分別 9/16/15 碼，符合此限制。

### 9.2 影像輸出品質（必須達成掃描可靠性）[REQ-BC-002]
條碼產生函式庫（Barcode Generation Library）輸出 PNG（或等效無損格式）時，必須符合：

1. **顏色/對比**：黑條白底（背景白色），禁止灰階漸層或彩色背景。
2. **解析度/尺寸策略（避免掃描失敗的關鍵）**  
   - RDLC 版面中條碼顯示尺寸固定；**產生影像時應以最終顯示尺寸與 DPI 直接生成**，避免 RDLC/印表機縮放造成條寬失真。  
   - 建議：以 **300 DPI 以上**等效解析度生成（或使用向量條碼輸出後轉點陣，但最終仍需在 PDF 中維持清晰邊界）。
3. **條碼高度**：建議落在 **8–14 mm**（以樣版視覺高度為準）；不得過矮。
4. **Quiet Zone（左右留白）**：左右需保留足夠空白，建議至少 **10×最窄條寬（X-dimension）**；若依超商作業建議，左右留白可設計為約 **2 cm**（以樣版為準）。
5. **寬窄比（wide:narrow）**：建議 **2:5** 或符合超商端掃描器常見容許範圍的比例（由函式庫參數固定，不得每筆浮動）。
6. **抗鋸齒**：避免因抗鋸齒導致邊界模糊。若函式庫可關閉 anti-aliasing 或提供 crisp mode，應啟用。
7. **最大長度限制**：單段條碼印出後長度不得超過 **6.2 cm**（以樣版/作業規範為準）。若版面造成超過，需調整 RDLC 版面或條碼模組寬度（X-dimension）。

### 9.3 三段條碼在樣版中的排版要求（與樣版一致）[REQ-BC-003]
1. 三段條碼由上而下排列。
2. 垂直間距：各段間距至少 **0.6 cm**（以樣版為準）。
3. 條碼區需搭配識別文字（例如「統一超商專用條碼區」）。若樣版含此文字，RDLC 需保留。

### 9.4 驗收標準（掃描可用性）[REQ-BC-004]
1. **螢幕預覽**：輸出 PDF 在 100% 縮放下條碼邊緣清晰、無糊邊、無斷線。
2. **列印測試（必做）**：使用實際將採用的印表機、紙張與列印設定列印 A4 成品。
3. **掃描成功率**：以超商讀碼設備/同等規格掃描器測試，三段條碼皆能在合理時間內成功讀取（建議目標：每段 < 1 秒、成功率 ≥ 99%）。  
4. **異常可追溯**：若任一段條碼生成失敗或掃描不穩定，Log 必須能定位（包含：該筆關鍵欄位、段別、字元集合檢核結果、影像尺寸/DPI 參數）。

## 11. 非功能性需求（NFR）

1. **一致性** [REQ-NFR-001]：同一筆資料在不同電腦輸出版面一致（RDLC 固定版面）
2. **可部署性** [REQ-NFR-002]：僅需部署 EXE + RDLC + 既有 `.accdb`（條碼函式庫以 NuGet 併入程式；不需安裝條碼字型）
3. **效能** [REQ-NFR-003]：1,000 筆資料匯出 PDF 於可接受時間內完成（依硬體訂 KPI）
4. **可靠性** [REQ-NFR-004]：任何錯誤必須回報明確原因（缺欄、連線失敗、條碼圖片生成失敗、列印/匯出失敗）
5. **可追溯性**（選配）[REQ-NFR-005]：可在 `.accdb` 寫入「列印紀錄」（時間、操作者、輸出檔名）

## 12. 驗收測試（AT）

### 12.1 版面驗收 [AT-001]
- 以 `3聯式繳費單樣版.pdf` 為對照，檢查：
  - 三聯位置、框線、虛線、標題/欄位對齊
  - 第三聯條碼區位置與大小符合樣版視覺
  - 一人一頁，不得跨頁、不漏頁

### 12.2 條碼可讀性驗收 [AT-002]
- 用至少 2 款超商常見條碼掃描器/App 測試（三段條碼皆可讀）
- 測試列印件（紙本）與 PDF 於不同電腦開啟的可讀性

### 12.3 資料一致性驗收 [AT-003]
- 報表上的每個欄位值必須與 `.accdb` 原始欄位一致（不做演算法重算）
- 條碼顯示字串必須為 `*{DB欄位}*`
